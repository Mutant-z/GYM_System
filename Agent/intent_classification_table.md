# Agent 意图分类表

本文档用于把用户输入划分为“业务类问题”和“知识类问题”，并进一步细分到可执行的意图，方便 `chat` 路由、工具调用、RAG 检索和网页知识补充。

## 1. 总体分流原则

| 一级分类 | 判定依据 | 主要处理方式 | 是否调用工具 | 是否查 RAG | 是否联网 |
|---|---|---|---:|---:|---:|
| 业务类问题 | 答案依赖系统实时业务数据，或用户希望 agent 执行操作 | `LLM + 工具调用 + Java 后端` | 是 | 否 | 否 |
| 知识类问题 | 答案依赖通用知识、平台说明、训练知识或文档内容 | `RAG + LLM`，必要时补网页知识 | 否 | 是 | 视情况 |

## 2. 业务类问题细分

业务类问题建议再拆成“查询类”和“操作类”。其中，查询类可以优先做只读工具，操作类必须走后端事务接口，不要让 agent 直接改数据库。

### 2.1 业务查询类

| 意图编码 | 意图名称 | 用户常见说法 | 需要的参数 | 处理方式 | 优先工具来源 | 示例回复目标 |
|---|---|---|---|---|---|---|
| `biz.booking.query` | 查询预约 | 我的预约、帮我查预约、我约了什么场地 | 用户身份、时间范围、场地类型 | LLM 识别参数后查业务数据 | Java 后端 API | 返回预约列表、状态、时间、场地 |
| `biz.booking.detail` | 查询预约详情 | 这条预约详情、具体是哪一个场地 | 预约 ID、日期、用户身份 | 调用详情查询接口 | Java 后端 API | 返回单条预约详情 |
| `biz.order.query` | 查询购买商品 | 我买过什么、我的订单、查询购买记录 | 用户身份、时间范围、订单状态 | 调用订单查询接口 | Java 后端 API | 返回订单列表、商品、金额、状态 |
| `biz.order.detail` | 查询订单详情 | 这笔订单详情、订单里有什么 | 订单 ID、用户身份 | 调用订单详情接口 | Java 后端 API | 返回订单明细 |
| `biz.course.enrollment.query` | 查询报名课程 | 我报了什么课、我的课程、查看报名记录 | 用户身份、时间范围、课程状态 | 调用课程报名查询接口 | Java 后端 API | 返回报名课程列表 |
| `biz.course.enrollment.detail` | 查询课程报名详情 | 这节课我什么时候上、报名信息 | 报名记录 ID、用户身份 | 调用报名详情接口 | Java 后端 API | 返回课程、时间、地点、状态 |
| `biz.member.status.query` | 查询会员状态 | 我还能不能预约、会员到期了吗 | 用户身份 | 调用会员状态接口 | Java 后端 API | 返回会员有效期、权限、状态 |
| `biz.gym.room.query` | 查询健身房/场地信息 | 还有空位吗、哪个房间可用 | 日期、时间段、场地类型 | 调用场地查询接口 | Java 后端 API | 返回可用房间、容量、时间段 |
| `biz.commodity.stock.query` | 查询商品库存 | 这个商品还有吗、库存多少 | 商品 ID、商品名 | 调用库存查询接口 | Java 后端 API | 返回库存、规格、是否可买 |
| `biz.course.query` | 查询课程信息 | 有什么课、课程详情、课程时间 | 课程 ID、课程名、时间范围 | 调用课程查询接口 | Java 后端 API | 返回课程列表、教练、时间 |

### 2.2 业务操作类

| 意图编码 | 意图名称 | 用户常见说法 | 需要的参数 | 处理方式 | 强制要求 | 示例回复目标 |
|---|---|---|---|---|---|---|
| `biz.booking.create` | 创建预约 | 帮我预约、我要订场地、预约今晚的房间 | 用户身份、场地、时间、人数 | LLM 补齐参数后调用创建接口 | 必须确认关键参数 | 返回预约是否成功、预约号 |
| `biz.booking.cancel` | 取消预约 | 帮我取消预约、退掉刚才的预约 | 预约 ID、用户身份 | 调用取消接口 | 必须校验归属和权限 | 返回取消结果、是否退款 |
| `biz.booking.modify` | 修改预约 | 改一下预约时间、换个房间 | 预约 ID、新时间、新场地 | 调用修改接口 | 必须校验冲突 | 返回修改结果 |
| `biz.course.enrollment.create` | 报名课程 | 帮我报名这节课、我要上课 | 课程 ID、用户身份 | 调用报名接口 | 必须检查名额和资格 | 返回报名成功与否 |
| `biz.course.enrollment.cancel` | 取消报名 | 取消课程报名、退课 | 报名记录 ID、用户身份 | 调用退课接口 | 必须检查退课规则 | 返回是否取消成功 |
| `biz.order.create` | 创建订单 | 帮我下单、买这个商品 | 商品 ID、数量、地址/配送方式 | 调用下单接口 | 必须校验库存和价格 | 返回订单号、金额 |
| `biz.order.cancel` | 取消订单 | 取消这个订单、撤销刚下的单 | 订单 ID、用户身份 | 调用取消订单接口 | 必须校验订单状态 | 返回取消结果 |
| `biz.member.update` | 修改会员资料 | 改手机号、更新个人信息 | 字段名、字段值、用户身份 | 调用资料修改接口 | 必须校验可修改字段 | 返回更新结果 |
| `biz.cart.add` | 加入购物车 | 加入购物车、先帮我放进去 | 商品 ID、数量 | 调用购物车接口 | 可选确认数量 | 返回加入成功与否 |
| `biz.cart.update` | 更新购物车 | 改购物车数量、删掉这个商品 | 购物车项 ID、数量/操作类型 | 调用购物车接口 | 必须校验商品状态 | 返回更新结果 |

### 2.3 业务决策规则

| 规则编号 | 规则说明 | 处理建议 |
|---|---|---|
| `biz.rule.01` | 只要问题涉及“我的/我已/我能否/帮我预约/帮我报名/帮我下单”，优先判定为业务类 | 优先走工具调用 |
| `biz.rule.02` | 只读查询可以直接返回结果，但也必须经过 LLM 组织语言 | 工具返回结构化数据，LLM 生成自然语言 |
| `biz.rule.03` | 写操作必须保留二次确认或参数确认步骤 | 先补齐关键参数，再执行 |
| `biz.rule.04` | 涉及权限、库存、冲突、时间窗、退费规则时，最终判断必须由后端完成 | agent 只做编排，不做最终裁决 |
| `biz.rule.05` | 如果业务工具失败，但问题仍是业务问题，不要自动转成知识问答 | 返回失败原因并提示重试或人工处理 |

## 3. 知识类问题细分

知识类问题建议拆成“内部知识”和“外部知识”两层。

### 3.1 内部知识类

| 意图编码 | 意图名称 | 用户常见说法 | 主要来源 | 处理方式 | 是否联网 |
|---|---|---|---|---|---:|
| `know.platform.rule` | 平台规则说明 | 课程怎么退、预约规则是什么、商品退款规则 | 内部文档、FAQ、规则说明 | RAG + LLM | 否 |
| `know.course.explain` | 课程内容说明 | 这节课是干什么的、适合谁 | 课程介绍、训练说明 | RAG + LLM | 否 |
| `know.equipment.use` | 器械使用说明 | 跑步机怎么用、器械怎么调 | 器械说明文档、场馆资料 | RAG + LLM | 否 |
| `know.faq.system` | 系统常见问题 | 为什么登录不了、怎么修改资料 | FAQ、帮助中心 | RAG + LLM | 否 |
| `know.policy.membership` | 会员政策说明 | 会员怎么续费、有哪些权益 | 会员制度文档 | RAG + LLM | 否 |

### 3.2 外部知识类

| 意图编码 | 意图名称 | 用户常见说法 | 主要来源 | 处理方式 | 是否联网 |
|---|---|---|---|---|---:|
| `know.fitness.basic` | 健身基础知识 | 深蹲怎么练、增肌怎么安排 | 知识库 + 网页知识 | RAG 优先，必要时联网补充 | 可能 |
| `know.training.plan` | 训练计划建议 | 一周怎么练、减脂怎么排课 | 知识库 + 网页知识 | RAG + 网页知识 + LLM | 可能 |
| `know.nutrition.basic` | 营养知识 | 蛋白粉什么时候喝、怎么吃更合适 | 知识库 + 网页知识 | RAG + 网页知识 + LLM | 可能 |
| `know.sports.science` | 运动科学常识 | 热身多久、拉伸有没有用 | 知识库 + 网页知识 | RAG + 网页知识 + LLM | 可能 |
| `know.latest.info` | 最新信息查询 | 今年最新训练趋势、最新研究 | 外部网页、新闻、公开资料 | 网页搜索 + LLM | 是 |

### 3.3 知识决策规则

| 规则编号 | 规则说明 | 处理建议 |
|---|---|---|
| `know.rule.01` | 如果问题不依赖用户个人数据，也不需要改系统状态，优先视为知识类 | 先查 RAG |
| `know.rule.02` | 如果内部知识库命中不足，再补网页知识 | 作为第二层来源 |
| `know.rule.03` | 如果涉及最新信息、标准变化、研究更新，优先联网 | 网页搜索后再回答 |
| `know.rule.04` | 知识类问题不要调用业务写接口 | 避免误操作 |
| `know.rule.05` | 如果 RAG 命中低且网页也不足，明确告诉用户信息不充分 | 不要编造 |

## 4. 混合型问题处理

有些问题既像业务问题，也像知识问题，需要拆解后再处理。

| 场景 | 处理顺序 | 说明 |
|---|---|---|
| “我这个会员能不能预约这节课，怎么安排更合适？” | 先查会员状态和课程信息，再给建议 | 前半句是业务，后半句是知识/建议 |
| “帮我看看这节课适不适合我报名” | 先查课程详情，再结合课程知识说明 | 先业务查询，再知识解释 |
| “我买的商品能不能退，退费规则是什么” | 先查订单，再查规则文档 | 前半句是业务，后半句是知识 |
| “帮我预约一个适合减脂的场地或课程” | 先查可用资源，再结合训练知识推荐 | 先业务筛选，再知识推荐 |

## 5. 推荐路由顺序

```text
用户输入
  -> LLM 意图识别
  -> 判断是否为业务类
     -> 是
        -> 业务查询 / 业务操作
        -> 调用 Java 后端工具
        -> LLM 组织答案
     -> 否
        -> 知识类
        -> 先查 RAG
        -> 命中不足则查网页知识
        -> LLM 组织答案
```

## 6. 设计建议

1. `chat` 路由只负责入口分流，不直接写业务逻辑。
2. `gym_tools` 只封装业务工具，不直接承担回答生成。
3. `rag_service` 只负责知识检索、上下文构造和知识问答。
4. 写操作一律交给 Java 后端，agent 不直接操作数据库。
5. 业务查询和知识问答的日志要分开记录，方便排查错误。
